## 一、质疑与假设

在开始设计技术情报雷达方案之前，我对题目本身的需求描述进行了分析，发现一些不够明确或存在潜在歧义的地方。为了保证方案既切实可行，又能够为内部决策提供可靠参考，我在此提出一些质疑，并列出每条质疑背后的假设。

**1. 资源限制不明确**

**质疑**：题目没有明确团队可用的算力、预算和基础设施条件。

**背后的假设**：假设团队资源有限，如果方案过于复杂或需要高算力，可能无法在 4 周内落地，且AI给出的“最优方案”不一定适配于现有资源。

**潜在风险**：技术情报雷达提出的高价值跟踪方向可能无法实际验证，导致建议不符合公司资源条件。

**2. 信息可靠性边界不明确**

**质疑**：系统输出可能存在不准确或 AI 幻觉信息。

**背后的假设**：假设 AI 生成内容可能存在错误，需要通过证据链、结构化字段和人工复核来验证信息可靠性。

**潜在风险**：内部决策者可能基于不准确的信息做出判断，影响决策质量。

**3. 评价标准不清晰**

**质疑**：题目中“值得投入资源”的判断标准没有明确说明。

**背后的假设**：假设不同人对“价值”的理解存在差异，短期收益与长期技术积累也可能冲突。如果没有统一标准，输出结果可能不一致。

**潜在风险**：不同模块输出的评估结果可能不一致，导致内部决策者误判优先级。

## 二、方案设计

### 2.1 总体架构

```text
信息源
  ↓
采集归一化
  ↓
证据画像
  ↓
知识层：知识图谱 + 向量库
  ↓
RAG 分析与多维评分
  ↓
专项 Agent 深挖
  ↓
报告复核
  ↓
知识库沉淀
  ↺ 人工反馈反向优化评分规则、知识图谱和检索结果
```

系统采用 **Python + 公开 API/RSS 采集 + 知识图谱 + 向量检索/RAG + 规则评分 + 轻量 Agent** 的技术组合。Python 负责采集、清洗、结构化处理和评分逻辑验证；公开 API/RSS 用于获取论文、代码、模型和社区动态；知识图谱用于表达“论文-代码-模型-作者-机构-任务-指标”的关系；向量库和 RAG 用于召回历史相似技术、过往 PoC 结论和内部评价；规则评分保证判断过程稳定、可解释；Agent 只用于专项深挖任务，不作为第一版系统的主控架构。

MVP 阶段的目标不是做一个全自动研究员，而是先跑通“发现-画像-检索-评分-复核-沉淀”的闭环。RAG、知识图谱和 Agent 会进入第一版设计，但采用轻量实现，避免把系统复杂度推高到 4 周内不可落地的程度。

### 2.2 关键模块说明

**信息源采集模块**

MVP 优先接入结构化程度高、工程价值强的信息源：

| 信息源 | 价值 |
| --- | --- |
| arXiv | 最新论文和技术路线 |
| Semantic Scholar | 论文元信息、引用数 |
| GitHub | 工程实现、Star、License、维护情况 |
| Hugging Face | 模型资产、下载量、License |
| Hacker News | 社区关注度、早期信号 |

Python 原型已实现这些来源的基础采集。若某个外部 API 超时或限流，系统会降级处理，继续使用其他来源返回的结果，避免单点失败。

**清洗归一化与技术条目合并**

不同平台返回的数据结构差异较大，因此系统会先统一成标准技术条目，包括来源、标题、链接、摘要、发布时间、作者、指标和标签等字段。

同一项技术可能同时出现在多个平台，例如论文在 arXiv，代码在 GitHub，模型在 Hugging Face，社区在 Hacker News 讨论。MVP 阶段先通过标题、链接和关键词做轻量合并；同时引入 embedding 相似度作为增强手段，用于识别标题不同但实质相关的论文、模型和代码仓库。LLM 只在低置信度合并场景中辅助判断，避免过度依赖模型自由发挥。

**证据画像模块**

系统不会只生成摘要，而是提取与决策相关的结构化证据：

- 是否有论文、代码、模型；
- 是否提到 Benchmark、数据集或实现细节；
- License 是否明确；
- GitHub Star、Hugging Face 下载量、论文引用数；
- 是否涉及医疗、金融、隐私等敏感领域；
- 是否存在高部署成本风险。

这一步尽量依靠可追溯字段和规则抽取完成，LLM 只用于补充摘要理解和技术点归纳。这样可以保证后续评分不是黑箱判断。

**知识图谱与 RAG 模块**

为了让系统不只是一次性打分器，MVP 会引入一个轻量知识层。知识图谱负责保存实体关系，例如：论文提出了什么方法、对应哪个 GitHub 仓库、是否有 Hugging Face 模型、属于哪个任务、使用了哪些 Benchmark、来自哪个机构或作者。第一版不需要上复杂图数据库，可以先用 SQLite/JSON 保存实体和关系，后续再迁移到 Neo4j 或图数据库。

RAG 模块负责把当前候选技术与历史知识连接起来。例如系统评估一篇新的 Agentic RAG 论文时，会从知识库中召回历史上类似的 RAG 框架、已做过的 PoC、失败原因和内部评价，再辅助生成“为什么值得/不值得继续跟进”的解释。这样系统不是孤立判断单个技术，而是在 Lab 自己的历史经验上做判断。

**多维评分与决策模块**

系统采用加权评分机制，而不是让 LLM 直接判断“值不值得跟进”。

| 维度 | 权重 | 说明 |
| --- | ---: | --- |
| 相关性 | 20% | 是否匹配 Lab 当前关注方向 |
| 工程可行性 | 25% | 是否有代码、模型、API、部署线索 |
| 可复现性 | 20% | 是否有可运行资产、Benchmark、数据集、实现细节 |
| 社区/学术热度 | 15% | GitHub Star、HF 下载量、引用量、社区讨论 |
| 新颖性 | 10% | 是否代表新方法、新框架或新路线 |
| 风险控制 | 10% | License、部署成本、合规、隐私等风险 |

其中工程可行性和可复现性合计 45%，因为该系统服务的是 Lab 投入决策，而不是单纯做学术综述。RAG 召回结果会作为评分解释和人工复核参考，但核心分数仍由结构化证据和权重规则计算。

系统输出四类行动建议：

| 分数区间 | 决策 | 含义 |
| --- | --- | --- |
| 80 分以上 | 立即跟进 | 建议投入 1-2 人做 PoC |
| 65-80 分 | 持续观察 | 进入候选池，补充证据后再决定 |
| 50-65 分 | 低优先级归档 | 保留记录，但暂不主动投入 |
| 50 分以下 | 暂不投入 | 当前证据不足或相关性不高 |

分数不是最终结论，而是资源排序工具。系统负责初筛，最终仍由研究员复核。

**专项 Agent 模块**

MVP 阶段不使用多 Agent 作为主流程，但会保留少量专项 Agent，用来处理边界清晰、可验证的任务：

- 论文阅读 Agent：提取贡献点、实验设置、Benchmark 和局限；
- 代码检查 Agent：读取 README、License、安装方式、依赖复杂度和最近维护情况；
- PoC 规划 Agent：根据技术条目生成 0.5-2 人天的小验证任务；
- 报告撰写 Agent：基于结构化证据和评分结果生成周报初稿。

这些 Agent 的输出必须绑定原始链接、证据字段或人工复核记录，不能直接作为最终决策。

**报告与反馈模块**

系统每周生成 Markdown 报告，包含 Top 技术条目、评分、理由、RAG 召回的相似案例、风险提示和建议动作。人工可以修改评分、补充判断、标记误判，并记录 PoC 结论。这些反馈会进入知识库，用于后续调整权重、优化 RAG 召回结果和补全知识图谱关系。

### 2.3 MVP 落地计划

考虑到 1-2 人、4 周内落地可用 MVP，我会控制范围，优先完成核心闭环：

- 第 1 周：完成信息源采集与统一数据结构，支持关键词扫描；
- 第 2 周：完成证据画像、轻量知识图谱和评分机制，输出多维评分；
- 第 3 周：接入向量检索/RAG，支持相似技术和历史 PoC 结论召回，生成 Markdown 周报；
- 第 4 周：加入人工复核与反馈机制，并为论文阅读、代码检查、PoC 规划配置轻量专项 Agent。

MVP 不追求复杂平台，而是先用 Python 原型保证核心链路可运行、可解释、可迭代。后续验证有效后，再封装为定时任务、Web 服务或内部工具平台。

## 三、关键设计决策

### 3.1 使用轻量 Agent，而不是全自动多 Agent

MVP 阶段不采用完全自主运行的多 Agent 架构。技术情报判断具有不确定性，一项技术是否值得跟进，不只取决于摘要和热度，还取决于 Lab 当前方向、算力条件、业务场景和已有技术栈。

因此第一阶段采用“规则主流程 + 专项 Agent 辅助”的方式。主流程负责采集、画像、RAG 召回和评分，保证稳定可控；Agent 只处理论文阅读、代码检查、PoC 规划和报告撰写等边界清晰的任务。这样既体现了 Agent 能力，又不会让系统成败依赖复杂编排。

### 3.2 使用 RAG 增强判断，但不让 RAG 直接决策

RAG 的价值在于把当前候选技术放到历史上下文中比较，而不是直接替代判断。例如评估一个新的 RAG 框架时，系统可以召回历史上已试用过的相似框架、PoC 结论、部署成本和失败原因，从而帮助研究员判断是否值得重复投入。

但 RAG 检索结果可能存在召回不全或相关性不足的问题，因此它只作为评分解释和人工复核依据。最终分数仍由结构化证据、权重规则和人工复核共同决定。

### 3.3 使用知识图谱沉淀技术关系

单纯列表式记录很难表达技术之间的关系，因此系统引入轻量知识图谱，用来沉淀“论文-模型-代码-作者-机构-任务-Benchmark-PoC 结论”的关系。这样后续不仅能看单个技术条目，还能观察某条技术路线的演进，例如 Agentic RAG、长上下文推理、多模态 Agent 等方向的关联项目和代表性证据。

MVP 阶段不强依赖复杂图数据库，而是先用 SQLite/JSON 保存实体与关系，确保 4 周内可落地。等数据规模扩大后，再升级到 Neo4j 或专门图数据库。

### 3.4 结构化评分，而不是让 LLM 直接判断

直接让 LLM 判断“是否值得跟进”虽然方便，但结果不稳定，也缺少可解释性。因此系统采用“结构化证据 + 加权评分 + RAG/LLM 辅助解释”的方式。

核心分数来自明确字段，例如是否有代码、模型、Star、下载量、是否命中 Lab 方向、是否存在风险等。LLM 更适合用于摘要生成、技术路线解释、相似方案比较和周报撰写，而不是直接替代评分规则。

### 3.5 优先考虑工程可行性

本题的真实场景是 AI Lab 内部决策，核心问题是“是否值得投入资源跟进”。因此系统不能只看论文新颖性或社区热度，而要优先判断是否能被实际验证。

工程可行性被设为最高权重，主要依据包括是否有开源代码、模型权重、API、SDK、CLI、Docker、部署说明，以及是否有足够社区使用迹象。这样可以避免系统推荐大量“看起来很新但无法落地”的内容。

### 3.6 Benchmark 和 License 不作为初筛硬门槛

在技术雷达初筛阶段，Benchmark、数据集和 License 信息很重要，但不应过早成为一票否决条件。很多高价值工程项目可能没有在简介中明确写 Benchmark，或者 License 需要二次确认，但这不代表它们没有跟进价值。

因此本方案将 Benchmark 设计为加分项，将 License 不明确设计为“待确认项”。只有在出现高部署成本、合规敏感、完全没有工程资产等情况时，系统才明显压低分数。

### 3.7 加入硬上限机制，防止热度误导

加权评分可能被单一高分维度拉偏。例如一个项目 Star 很高，但与 Lab 当前方向无关；或者一篇论文很新颖，但没有代码、模型，也没有复现实验。

因此系统加入硬上限机制：相关性不足时不能进入高优先级；没有代码或模型时不能直接进入高优先级 PoC；部署成本明显过高或涉及合规敏感时必须进入人工复核。这可以防止系统被“热度”或“新颖性”单独误导。

### 3.8 先用 Python 原型落地

考虑到 4 周内由 1-2 人完成，MVP 应优先保证核心链路跑通。Python 原型开发快，适合接入公开 API、处理结构化数据、调试评分逻辑，并输出 JSON 或 Markdown 报告。

当前原型已经能完成多信息源采集、技术条目归一化、证据画像生成、多维评分和决策建议输出。下一步可在原型中补充 SQLite/JSON 知识层、向量检索和轻量 Agent 任务封装。后续如果验证有效，再将其封装为定时任务、Web 服务或内部工具平台。
